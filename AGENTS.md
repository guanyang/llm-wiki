# AGENTS.md — LLM Wiki Schema

> 本文件是 LLM 维护 Wiki 的元配置。LLM 必须在每次操作前读取本文件，严格遵守以下规范。

---

## 架构概览

本知识库采用四层架构：

| 层 | 位置 | 所有者 | 权限 |
|---|---|---|---|
| Schema | `AGENTS.md` | 人类 + LLM 共同演进 | LLM 只读 |
| Raw Sources | `raw/` | 人类 | LLM 只读，绝不修改 |
| Wiki | `wiki/` | LLM | LLM 读写，人类只读 |
| Output | `output/` | LLM 生成，人类审核定稿 | LLM 读写 |

## 目录结构

```
├── AGENTS.md              # 本文件——架构规范（人类与 LLM 共同演进）
├── raw/                   # 原始素材——不可变，人类策展
│   ├── articles/          # 网络文章、博客（markdown）
│   ├── papers/            # 学术论文、白皮书
│   ├── docs/              # 官方文档摘录
│   ├── transcripts/       # 会议记录、演讲稿、播客笔记
│   └── assets/            # 图片、图表、数据文件
├── wiki/                  # LLM 编译产物——LLM 完全拥有此层
│   ├── index.md           # 全局索引——所有页面的目录
│   ├── log.md             # 操作日志——按时间顺序记录
│   ├── entities/          # 实体页（工具、框架、组织、人物）
│   ├── concepts/          # 概念页（设计模式、方法论、理论框架）
│   ├── summaries/         # 素材摘要页（每份已消化的素材一个文件）
│   ├── comparisons/       # 对比分析页（A vs B 式对比）
│   └── synthesis/         # 综合分析页（跨素材综合、趋势判断、个人洞察）
└── output/                # 成品输出——从 wiki 提炼的可交付内容
    ├── posts/             # 博客文章、公众号文章
    ├── reports/           # 研究报告、技术调研报告
    ├── slides/            # 演示文稿（Marp markdown 格式）
    ├── tutorials/         # 教程、手把手指南
    └── newsletters/       # 周报、月报、知识简报
```

### raw/ — 原始资料层

此层是知识的源头，**不可变**——LLM 只读取，永远不修改。人类负责策展和维护。

| 子目录 | 用途 | 存放内容 | 格式建议 |
|--------|------|---------|---------|
| `raw/articles/` | 网络文章与博客 | 通过 Obsidian Web Clipper 或手动保存的文章、技术博客、新闻报道 | Markdown（`.md`） |
| `raw/papers/` | 学术论文与白皮书 | 研究论文、行业白皮书、技术报告。可存放 PDF 原件和对应的文本提取版 | PDF（`.pdf`）+ Markdown 提取（`.md`） |
| `raw/docs/` | 官方文档摘录 | 工具/框架/平台的官方文档片段、API 参考、配置说明 | Markdown（`.md`） |
| `raw/transcripts/` | 会议与演讲记录 | 会议纪要、技术演讲文字稿、播客笔记、视频课程笔记、读书笔记 | Markdown（`.md`） |
| `raw/assets/` | 媒体与数据文件 | 架构图、流程图、截图、数据表格等被其他资料引用的附件 | PNG/SVG/CSV 等 |

**提示**：
- 每份资料建议在文件开头注明来源 URL、作者、日期，方便后续溯源。
- 文件命名使用 `YYYY-MM-DD-简短描述.md` 格式（如 `2026-04-10-playwright-vs-cypress.md`），便于按时间排序。
- `assets/` 中的文件通过相对路径被其他资料引用，避免使用外部 URL（链接可能失效）。

### wiki/ — LLM 编译产物

此层由 **LLM 完全拥有和维护**。人类只读取浏览，不直接编辑。所有变更通过与 LLM 对话触发。

| 文件/子目录 | 用途 | 说明 |
|------------|------|------|
| `wiki/index.md` | 全局索引 | 所有 wiki 页面的目录，按类别组织，附带一行摘要。LLM 每次消化素材时更新。查询时 LLM 首先阅读此文件定位相关页面。 |
| `wiki/log.md` | 操作日志 | 按时间顺序的追加式记录——消化、查询、巡检等操作。每条记录使用统一前缀格式，可用 `grep` 等工具解析。 |
| `wiki/entities/` | 实体页 | 每个具体实体一个文件（如工具、框架、平台、组织、人物、项目）。记录事实性信息、特征、优劣势。 |
| `wiki/concepts/` | 概念页 | 每个抽象概念一个文件（如设计模式、方法论、理论框架）。解释概念本身，交叉引用相关概念和实体。 |
| `wiki/summaries/` | 素材摘要页 | 每份已消化的原始素材一个文件。包含摘要、关键要点、与现有知识的关联。是 `raw/` 中素材的"编译产物"。 |
| `wiki/comparisons/` | 对比分析页 | A vs B 式的工具对比、方案评估。通常由查询触发生成，也可在消化素材时主动创建。 |
| `wiki/synthesis/` | 综合分析页 | 跨多份素材的综合分析、趋势判断、个人洞察、操作指南、知识库总览等。承载需要融合多个来源才能产出的高价值内容。 |

### output/ — 成品输出层

从 wiki 中提炼的面向外部受众的可交付内容。LLM 协助生成，人类审核定稿。

| 子目录 | 用途 | 说明 |
|--------|------|------|
| `output/posts/` | 博客文章 | 博客、公众号文章等公开发表的长文。从 wiki 多个页面提炼，独立可读，不依赖 `[[双链]]`。 |
| `output/reports/` | 研究报告 | 技术调研报告、竞品分析报告等结构化文档。包含摘要、正文、结论、参考文献。 |
| `output/slides/` | 演示文稿 | Marp 格式的 Markdown 幻灯片。可直接转换为 PDF/HTML 演示。 |
| `output/tutorials/` | 教程指南 | 面向读者的手把手教程。步骤清晰、代码完整、可独立跟随操作。 |
| `output/newsletters/` | 知识简报 | 周报、月报等定期简报。汇总近期消化的素材和关键洞察。 |

**与 wiki 的区别**：
- wiki 页面面向自己，持续演进，使用 `[[双链]]`
- output 成品面向他人，定稿后基本不变，独立可读，使用标准 Markdown 链接

## 页面格式规范

每个 wiki 页面必须包含 YAML frontmatter：

```yaml
---
title: 页面标题
aliases: [别名1, 别名2]
tags: [标签1, 标签2]
category: entities | concepts | summaries | comparisons | synthesis
created: YYYY-MM-DD
updated: YYYY-MM-DD
sources:
  - "[[raw/路径/源文件名]]"
description: 一句话摘要，不超过100字
---
```

### 标签体系

tags 字段从以下三个维度选取，根据需要组合，保持一致性：

**领域标签**（页面涉及的技术/业务领域）：
`AI`、`软件工程`、`测试`、`CI-CD`、`基础设施`、`可观测性`、`前端`、`后端`、`DevOps`、`架构`、`编程语言`、`知识管理`、`数据库`、`中间件`、`分布式`、`云原生`、`安全`、`信息科学`

**类型标签**（页面描述的事物类型）：
`框架`、`工具`、`模式`、`实践`、`标准`、`概念`、`人物`、`项目`、`方法论`、`对比分析`、`综合分析`

**成熟度标签**（可选，仅在明确时标注）：
`成熟`、`新兴`、`实验性`、`已废弃`

**使用规则：**
- 每个页面至少包含 1 个领域标签和 1 个类型标签
- 成熟度标签可选，仅在有明确判断依据时标注
- 优先使用已有标签，新增标签前先检查是否有近义词可复用
- 标签用中文，保持与 wiki 整体语言一致

### 正文结构要求
- 使用清晰的 Markdown 标题层级（h2、h3）
- 关键术语使用 `[[双链]]` 引用其他 wiki 页面
- 引用原始素材时使用 `[[raw/路径/文件名]]` 格式
- 每个页面末尾包含 `## 相关页面` 章节，列出所有交叉引用
- 代码块标注语言类型

## index.md 规范

`wiki/index.md` 是全局索引，LLM 每次操作后必须更新。

格式：

```markdown
# Wiki Index

> 最后更新：YYYY-MM-DD | 页面总数：N | 素材总数：M

## Entities
- [[Obsidian]] - 基于本地 markdown 的知识管理工具 (sources: 1)

## Concepts
- [[RAG]] - 检索增强生成，LLM + 文档交互的主流方式 (sources: 1)

## Summaries
- [[raw-articles-llm-wiki]] - Karpathy 的 LLM Wiki 模式完整阐述 (2026-04-12)

## Comparisons
- [[LLM-Wiki-vs-RAG]] - LLM Wiki 模式与传统 RAG 的对比分析 (sources: 1)

## Synthesis
- [[index与log的扩展性分析]] - index.md 和 log.md 随规模增长的瓶颈及分阶段应对策略
```

## log.md 规范

`wiki/log.md` 是操作日志，仅追加，不修改历史记录。

格式：

```markdown
# Wiki Log

## [YYYY-MM-DD] ingest | 素材标题
- 来源：`raw/路径/文件名`
- 新建：[[页面1]]、[[页面2]]
- 更新：[[页面3]]（新增xxx章节）
- 更新：index.md

## [YYYY-MM-DD] query | 查询问题
- 查阅：[[页面1]]、[[页面2]]
- 产出：[[对比分析页]]（已存入 wiki）

## [YYYY-MM-DD] lint | 健康检查
- 发现矛盾：[[页面A]] 与 [[页面B]] 关于xxx的描述不一致
- 孤立页面：[[页面C]]（无入链）
- 缺失概念页：建议创建 [[概念X]]
- 已修复：N 项 | 待修复：M 项

## [YYYY-MM-DD] publish | 成品标题
- 类型：post | report | slides | tutorial | newsletter
- 输出：`output/posts/文件名.md`
- 来源：[[wiki 页面1]]、[[wiki 页面2]]、...
- 状态：草稿 | 已审核
```

## 工作流程

### Ingest（摄入）

当用户要求处理新素材时，严格按以下步骤执行：

1. **读取源文件**：从 `raw/` 中读取指定文件，理解全文内容
2. **提取关键信息**：识别核心观点、实体、概念、数据、结论
3. **与用户讨论要点**：向用户汇报关键发现，确认重点和方向
4. **创建/更新摘要页**：在 `wiki/summaries/` 创建该素材的摘要页
5. **更新相关实体页**：在 `wiki/entities/` 中创建新实体页或更新已有实体页，补充新信息
6. **更新相关概念页**：在 `wiki/concepts/` 中创建新概念页或更新已有概念页
7. **维护交叉引用**：确保所有相关页面之间有 `[[双链]]` 互相引用
8. **更新 index.md**：在 `wiki/index.md` 中添加或更新条目
9. **记录日志**：在 `wiki/log.md` 追加本次操作记录

**注意事项：**
- 如果新素材与已有 wiki 内容矛盾，在相关页面中明确标注矛盾，注明新旧来源
- 单次 ingest 可能触发 10-15 个页面的更新，这是正常的
- 绝不修改 `raw/` 中的任何文件

### Query（查询）

当用户提出问题时，按以下步骤执行：

1. **读取 index.md**：浏览全局索引，定位可能相关的页面
2. **读取相关页面**：深入阅读相关的 entity、concept、summary 等页面
3. **合成答案**：基于 wiki 内容综合回答，而非从原始素材重新推导
4. **提供引用**：答案中引用具体的 wiki 页面和原始素材来源
5. **询问存档**：如果答案具有复用价值（对比分析、综合洞察），询问用户是否将其存入 `wiki/comparisons/` 或 `wiki/synthesis/`

**注意事项：**
- 优先使用 wiki 中已编译的知识，而非每次回到 raw 重新检索
- 如果 wiki 中信息不足，可以回溯 `raw/` 补充，并顺便更新 wiki

### Lint（健康检查）

当用户要求检查 wiki 健康状况，或 LLM 主动建议时，执行以下检查：

1. **矛盾检测**：检查页面之间是否存在相互矛盾的声明
2. **孤立页面**：查找没有任何入链的页面（index.md 中的链接不算）
3. **缺失概念页**：识别被多次 `[[引用]]` 但尚未创建的页面
4. **过时声明**：检查是否有被更新素材取代的旧结论
5. **交叉引用完整性**：确保相关页面之间有双向链接
6. **生成修复建议**：输出问题清单，与用户确认后执行修复
7. **记录日志**：在 `wiki/log.md` 追加 lint 记录

### Publish（发布）

当用户要求从 wiki 中提炼成品输出时，按以下步骤执行：

1. **明确需求**：确认输出类型（博客/报告/幻灯片/教程/简报）、目标受众、篇幅要求
2. **定位素材**：读取 `wiki/index.md`，找到所有相关的 wiki 页面
3. **阅读来源**：深入阅读相关的 entity、concept、comparison、synthesis 页面
4. **生成草稿**：基于 wiki 内容撰写成品，存入 `output/` 对应子目录
5. **格式适配**：
   - 将 `[[双链]]` 转换为标准 Markdown 链接或纯文本（成品必须独立可读）
   - 博客/报告：标准 Markdown，包含标题、摘要、正文、参考
   - 幻灯片：Marp 格式（`---` 分页，frontmatter 含 `marp: true`）
   - 简报：简洁的条目式结构
6. **用户审核**：将草稿呈现给用户，根据反馈修改，直到用户满意
7. **记录日志**：在 `wiki/log.md` 追加 publish 记录

**成品 frontmatter 格式**：

```yaml
---
title: 成品标题
type: post | report | slides | tutorial | newsletter
audience: 目标受众描述
created: YYYY-MM-DD
status: draft | reviewed | published
wiki_sources:
  - "wiki/页面路径"
---
```

**注意事项**：
- 成品必须独立可读，不依赖 wiki 内部的 `[[双链]]`
- 引用来源时使用脚注或参考文献格式
- LLM 生成草稿，人类审核定稿——output 层是协作产物，不是 LLM 单方面拥有
- 简报类输出可以半自动化：LLM 读取 `wiki/log.md` 近期记录，自动汇总

## 命名约定

| 类型 | 命名规则 | 示例 |
|---|---|---|
| 实体页 | 实体名称，中划线分隔 | `Obsidian.md` |
| 概念页 | 概念名称 | `RAG.md` |
| 摘要页 | `raw-` + 源路径简写 | `raw-articles-llm-wiki.md` |
| 对比页 | `A-vs-B-主题` | `LLM-Wiki-vs-RAG.md` |
| 综合页 | 分析主题描述 | `index与log的扩展性分析.md` |

## 核心原则

1. **Raw 不可变**：绝不修改 `raw/` 下的任何文件，这是事实来源
2. **Wiki 是编译产物**：wiki 中的所有内容都可以从 raw 重新生成
3. **知识复利**：每次 ingest 和 query 都让 wiki 变得更丰富，知识持续积累
4. **交叉引用优先**：页面之间的链接和引用是 wiki 最大的价值
5. **矛盾透明**：发现矛盾不要隐藏，明确标注并注明来源
6. **日志完整**：所有操作都记录在 log.md，保持可追溯性
7. **渐进演化**：本 Schema 会随使用经验持续优化，由人类和 LLM 共同维护
